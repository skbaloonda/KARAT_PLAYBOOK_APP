System Design Principles and Patterns

Master the core principles and common patterns for designing scalable, performant frontend systems.

Componentization

Break UI into reusable, maintainable components

Example:

Create atomic components like Button, Card, Input that can be composed into larger features.

State Management

Use predictable patterns (Redux, Context API, RxJS)

Example:

Centralize application state with Redux for complex apps or Context API for simpler needs.

Performance Optimization

Lazy loading, code splitting, virtualization

Example:

Implement React.lazy() for route-based code splitting and virtualization for large lists.

Accessibility (A11y)

Semantic HTML, ARIA roles, keyboard navigation

Use semantic elements, add ARIA labels, ensure all interactions work with keyboard only.

Security

Clarify Requirements

"Should the dashboard support mobile devices? What's the expected user load?"

2

Users, concurrent sessions, expected load on UI

Example:

"Expecting 100K concurrent users, 50K data points on charts."

3

High-Level Architecture

Show page structure, routing, state management, API integration

4

5

Example:

"Implement Service Workers for offline support, error boundaries for graceful failures."

Practice Questions

Practice questions for frontend system design interviews.

Priority Questions

1

What are hooks in React?

Answer: Hooks are functions that let you use React features (state, lifecycle, context) in functional components — e.g., useState, useEffect, useContext. They enable reusable logic without classes.
2

Explain the useEffect hook in React. What are its typical use cases?

Answer: useEffect runs side-effects after render (data fetching, subscriptions, DOM updates). Typical uses: fetching data on mount, subscribing/unsubscribing to events, updating document title, and syncing props to state. Cleanup function handles teardown.
3

What is forwardRef in React, and when would you use it?

Answer: forwardRef lets a parent pass a ref through a component to access a DOM node or child instance. Use it to expose focus/scroll methods or integrate with third‑party libraries that need DOM refs.
4

What are controlled and uncontrolled components in React?

Answer: Controlled components have their value driven by React state (value + onChange). Uncontrolled components manage their own state via the DOM (refs). Controlled gives predictable state flow; uncontrolled can be simpler for simple inputs.
5

Provide two examples of scenarios where controlled and uncontrolled components are appropriate.

Answer: Controlled: complex forms requiring validation, realtime UI updates (e.g., live preview). Uncontrolled: simple one-off inputs or third-party widgets where direct DOM access is easier (e.g., file inputs, quick demos).
6

What is useState in React, and how is it used?

Answer: useState is a Hook that adds state to functional components. Call const [state, setState] = useState(initial) and update via setState. It schedules re-renders and preserves state between renders.
7

How can we optimize React components to avoid unnecessary re-renders?

Answer: Techniques: memoize components with React.memo, useMemo/useCallback for expensive computations and handlers, avoid creating new props/objects inline, split components, use immutable updates, and key lists correctly.
8

How can we use the useRef hook in React?

Answer: useRef returns a mutable ref object. Use it to access DOM nodes, store mutable values that persist across renders without causing re-renders (e.g., timers), or keep previous values.
9

What is the difference between useCallback and useMemo in React?

Answer: useCallback memoizes a function reference; useMemo memoizes a computed value. useCallback(fn, deps) returns the same function between renders; useMemo(() => value, deps) caches the result of an expensive calculation.
10

How can we pass data from a child component to a parent component in React?

Answer: The common pattern is to pass a callback from parent to child as a prop (e.g., onChange). The child calls the callback with data, and the parent updates its state accordingly. For complex cases, use context or global state stores.

Mid-Level Questions

1

Explain the box model and how to center an element horizontally and vertically.

Answer: Box model: content, padding, border, margin. Center horizontally with margin:auto or flexbox; vertically with flexbox (align-items:center; justify-content:center) or grid and align-content.
2

How does CSS specificity work?

Answer: Specificity is calculated from inline styles, IDs, classes/attributes/pseudo-classes, and elements. Higher specificity wins; use !important sparingly.
3

Explain event delegation and why it is useful.

Answer: Attach a single handler on a parent and use event.target to handle child events; reduces listeners and supports dynamic elements.
4

What is the virtual DOM and why use it?

Answer: Virtual DOM is an in-memory representation of UI; diffing minimizes real DOM updates for performance in frameworks like React.
5

Describe how to optimize web performance.

Answer: Optimize assets (minify, compress), use caching/CDN, lazy-load images, reduce render-blocking CSS/JS, and use performance budgets and measuring tools.
6

Explain CSS Flexbox vs Grid and when to use each.

Answer: Flexbox is 1D (row/column) for alignment; Grid is 2D for layout with rows/columns. Use flexbox for components, grid for page layouts.
7

How do you handle cross-browser compatibility?

Answer: Use feature detection, polyfills, progressive enhancement, vendor prefixes, and test across major browsers and devices.
8

What is CORS and how to resolve CORS issues?

Answer: CORS controls cross-origin requests; resolve by setting Access-Control-Allow-Origin on server or using proxy; configure allowed methods and headers.
9

Explain immutability in JavaScript and benefits.

Answer: Immutability means not modifying existing objects; benefits include easier reasoning, fewer side effects, and simpler change detection in state management.
10

How do you prevent XSS on a web page?

Answer: Sanitize/escape user input, use Content Security Policy, avoid dangerous innerHTML, and validate on server-side.
11

Describe how to implement responsive images.

Answer: Use srcset and sizes, picture element, and serve appropriately sized and compressed images depending on device pixel ratio and viewport.
12

What are service workers and when to use them?

Answer: Service workers run in background to enable offline caching, push notifications, and background sync; use for PWAs and offline-first apps.
13

Explain debouncing vs throttling.

Answer: Debounce delays until input stops; throttle limits calls to a maximum rate. Use debounce for search inputs, throttle for scroll/resize handlers.
14

How do you manage state in a large frontend app?

Answer: Use component-local state for UI, lift state up, and centralized stores (Redux, MobX, Context) for cross-cutting state with clear patterns and immutability.
15

What is tree shaking?

Answer: Tree shaking removes unused exports during bundling (e.g., with ES modules) to reduce bundle size.
16

Explain how HTTP/2 improves performance over HTTP/1.1.

Answer: HTTP/2 supports multiplexing, header compression, server push, and binary framing to reduce latency and improve resource loading.
17

How do you implement accessibility in forms?

Answer: Use semantic HTML, labels, aria attributes where needed, keyboard navigation, and test with screen readers and contrast checks.
18

What are CSS custom properties and how do they help?

Answer: CSS variables allow reuse and runtime theming, reduce duplication, and can be updated via JS for dynamic styles.
19

Explain the differences between GET and POST requests.

Answer: GET retrieves resources and is idempotent, data in URL; POST submits data, can change server state, and sends body payloads.
20

How do you measure and improve First Contentful Paint (FCP)?

Answer: Measure with Lighthouse/CrUX, reduce render-blocking CSS/JS, inline critical CSS, optimize fonts, and server-side rendering or preloading critical assets.

Senior-Level Questions

1

Design a front-end architecture for a large-scale web app.

Answer: Use modular component architecture, micro-frontends if needed, enforce conventions, centralize state and routing, and optimize build and deployment pipelines for independent teams.
2

How would you reduce bundle size in a matured application?

Answer: Code-splitting, lazy-load routes, remove unused dependencies, enable tree shaking, analyze bundles and replace heavy libs with lighter alternatives.
3

Explain strategies for front-end observability.

Answer: Capture metrics (RUM), logs, distributed traces, error reporting, user sessions, and correlate client metrics with backend traces for end-to-end visibility.
4

How do you drive performance SLAs for web apps?

Answer: Define SLOs (FCP, LCP, TTFB), monitor with synthetic and real-user metrics, set alerts and runbooks, and prioritize engineering work to meet SLAs.
5

Describe migration plan to micro-frontends.

Answer: Start with boundaries, extract teams’ areas into independent deployable units, standardize contracts, handle shared dependencies, and roll out incrementally with feature flags.
6

How to ensure security in a single-page app?

Answer: Protect APIs, use safe token storage, validate inputs on server, apply CSP, use secure headers, and perform threat modeling and dependency audits.
7

Explain strategies for CSS at scale.

Answer: Use CSS modules, design tokens, utility classes, enforce naming conventions, and adopt a theming system to reduce global conflicts.
8

How would you approach accessibility for a complex web app?

Answer: Integrate accessibility into design system, automate checks, include manual testing with screen readers, and train teams on ARIA best practices.
9

Describe progressive hydration and its benefits.

Answer: Progressive hydration hydrates interactive parts incrementally to improve TTI; it reduces main-thread work and speeds perceived interactivity.
10

How do you mentor junior engineers on front-end best practices?

Answer: Pair-program, review code with constructive feedback, teach patterns and testing, and provide clear docs and examples in a design system.
11

Explain client-side caching strategies.

Answer: Use HTTP cache headers, service worker caching strategies (stale-while-revalidate), IndexedDB for complex offline cache, and cache invalidation strategies.
12

How would you architect real-time features in a front-end?

Answer: Use WebSockets or WebRTC for low-latency updates, fall back to polling, design event schemas, and debounce/display UX to avoid jitter.
13

What are key trade-offs of server-side rendering vs client-side rendering?

Answer: SSR improves SEO and initial paint but increases server complexity; CSR offers rich client interactivity and reduces server load after first render.
14

Discuss approaches for large-scale component libraries versioning.

Answer: Semantic versioning, backward-compatible deprecations, migration guides, and automated compatibility tests across consuming apps.
15

How do you design APIs that front-end teams consume?

Answer: Keep API contract stable, provide pagination/filtering, document with OpenAPI, version deliberately, and consider GraphQL for flexible queries.
16

How do you approach A/B testing front-end features?

Answer: Define clear metrics, use feature flags, run randomized experiments, collect telemetry, and analyze statistical significance before rollout.
17

Explain how to instrument front-end code for performance debugging.

Answer: Add RUM metrics, custom marks/measures, use Performance API, capture traces for slow user flows, and correlate with backend logs.
18

What are the considerations for internationalization (i18n)?

Answer: Externalize strings, support pluralization/RTL, lazy-load locale bundles, and ensure date/number formatting is locale-aware.
19

How do you evaluate a third-party frontend library for adoption?

Answer: Check maintenance, size, license, community, compatibility, security, and whether it fits the long-term roadmap and performance needs.
20

Describe strategies to reduce frontend technical debt.

Answer: Regular refactoring, code reviews enforcing standards, automated tests, incremental rework with feature flags, and tracking tech debt in backlog.

Scenario 1: Design a Responsive E-Commerce Product Page

Interviewer:

Imagine you need to design a responsive product page for an e-commerce site. How would you approach this?

Candidate:

First, I'll clarify requirements: the page should display product details, images, pricing, and support mobile and desktop views. Non-functional requirements include fast load time and SEO optimization.

Interviewer:

How would you structure the components?

Candidate:

I'd break the page into reusable components: ProductImage, ProductDetails, PriceTag, and AddToCartButton. For layout, I'd use CSS Grid or Flexbox for responsiveness. State management can be local since it's mostly static data, but if cart state is global, I'd use Context API or Redux.

Performance optimization: I'd implement lazy loading for images, compress assets, and use responsive image tags (srcset). For SEO, I'd ensure semantic HTML and server-side rendering if needed.

Scenario 2: Autocomplete Search Component

Interviewer:

Design an autocomplete search component for a large dataset.

Candidate:

I'd start with an input field that triggers API calls on user input. To avoid excessive calls, I'd debounce input (e.g., 300ms). For accessibility, I'd use ARIA roles and keyboard navigation.

Performance: I'd cache recent queries in memory and limit results to top 10. For large datasets, I'd implement pagination or infinite scroll.

Scenario 3: Dashboard for Real-Time Analytics

Interviewer:

Design a dashboard that shows real-time analytics updates.

Candidate:

I'd use a component-based architecture: Chart, StatsCard, and Notifications. For real-time updates, I'd use WebSockets or SSE. State management via Redux for predictable updates. Performance optimization includes virtualization for large tables and memoization for charts.

Scenario 1: Micro-Frontend Architecture

Interviewer:

How would you design a micro-frontend architecture for a large enterprise app?

Candidate:

I'd split the app into independent micro-frontends per domain (e.g., Payments, Profile). Each team owns its build pipeline. For integration, I'd use Module Federation or an iframe-based approach. Shared libraries like design systems would be versioned and distributed via npm.

Interviewer:

How do you handle routing and shared state?

Candidate:

Routing can be centralized using a shell app. Shared state can be managed via a global event bus or a lightweight store like Redux Toolkit.

Scenario 2: Real-Time Collaborative Document Editor

Interviewer:

Design a real-time collaborative editor like Google Docs.

Candidate:

I'd use WebSockets for real-time sync and implement Operational Transformation or CRDT for conflict resolution. For offline edits, I'd queue changes locally and sync when online. Rendering optimization includes batching updates and using virtualization for large documents.

Security: I'd use JWT for authentication, encrypt data in transit (TLS), and sanitize user input to prevent XSS.

Scenario 3: Global E-Commerce Frontend

Interviewer:

Design a frontend for a global e-commerce platform with localization.

Candidate:

I'd implement i18n using libraries like react-intl or Angular i18n. For performance, I'd prefetch locale-specific bundles and use CDN edge caching. Accessibility includes proper language attributes and RTL support.

Core Web Vitals: I'd use code splitting, lazy loading, and image optimization. Monitoring via Lighthouse and real-user metrics (LCP, CLS, FID).
